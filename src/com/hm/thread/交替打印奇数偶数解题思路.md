# 交替打印奇数和偶数 解题文档

## 一、题目描述

启动两个线程：一个线程只负责打印奇数，另一个线程只负责打印偶数，
两个线程**交替**输出，最终按自然数顺序 `1, 2, 3, 4, ...` 打印到指定上限 `maxNumber`。

**示例（maxNumber = 6）：**

| 线程 | 输出 |
|------|------|
| Odd-Thread  | 1 |
| Even-Thread | 2 |
| Odd-Thread  | 3 |
| Even-Thread | 4 |
| Odd-Thread  | 5 |
| Even-Thread | 6 |

要求：输出严格交替，不能出现两个奇数（或两个偶数）连续打印，也不能越界打印大于 `maxNumber` 的值。

---

## 二、核心观察 / 关键点

1. **共享状态**：两个线程共享同一个计数器 `number` 和上限 `maxNumber`，对它的读写必须在同一把锁/同一同步机制下进行，否则有可见性与竞态问题。
2. **交替的本质是“轮流让出执行权”**：奇数线程打印完一次必须把执行权交给偶数线程，反之亦然。这是一个典型的**线程通信（生产-消费式信号传递）**问题。
3. **必须用 `while` 循环判断条件，而不是 `if`**：被唤醒后要重新检查“是否真的轮到我”，防止虚假唤醒（spurious wakeup）以及多线程下的条件失效。
4. **收尾要唤醒对方**：当计数到达上限、本线程退出循环时，要 `notify/signal` 对方一次，否则对方可能永久阻塞在 `wait/await` 上，导致程序无法结束。

---

## 三、三种解法

### 解法一：synchronized + wait/notify

**思路：**
1. 用一个对象 `lock` 作为监视器，两个线程进入各自方法时都 `synchronized (lock)`。
2. 奇数线程循环检查：若当前 `number` 是偶数（没轮到自己），调用 `lock.wait()` 释放锁并阻塞；否则打印、`number++`、`notifyAll()` 唤醒对方。
3. 偶数线程逻辑对称。
4. 退出循环后再 `notifyAll()` 一次，避免对方卡死。

**关键代码：**

```java
public void printOdd() {
    synchronized (lock) {
        while (number <= maxNumber) {
            if (number % 2 == 0) {
                lock.wait();              // 没轮到奇数，让出锁等待
            } else {
                System.out.println("Odd Thread: " + number);
                number++;
                lock.notifyAll();         // 唤醒偶数线程
            }
        }
        lock.notifyAll();                 // 收尾唤醒，防止对方永久阻塞
    }
}
```

- **时间复杂度：** O(n)，n = maxNumber
- **空间复杂度：** O(1)
- **优点：** JDK 最基础的同步原语，无需额外类，面试高频
- **缺点：** `notifyAll` 会把所有等待线程都唤醒（本题只有 2 个线程影响不大）；语义不如 Condition 精确

---

### 解法二：ReentrantLock + Condition（精确唤醒）

**思路：**
1. 用一把 `ReentrantLock`，为奇、偶线程各建一个 `Condition`：`oddCondition`、`evenCondition`。
2. 奇数线程：没轮到时 `oddCondition.await()`；打印后 `evenCondition.signal()` **精确唤醒**偶数线程。
3. 偶数线程对称。
4. `await/signal` 必须在 `lock.lock()` 与 `unlock()` 之间，`unlock()` 放在 `finally` 保证释放。

**关键代码：**

```java
public void printOdd() {
    lock.lock();
    try {
        while (number <= maxNumber) {
            if (number % 2 == 0) {
                oddCondition.await();       // 奇数线程在自己的队列等待
            } else {
                System.out.println("Odd Thread: " + number);
                number++;
                evenCondition.signal();     // 精确唤醒偶数线程
            }
        }
        evenCondition.signal();             // 收尾唤醒对方
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        lock.unlock();
    }
}
```

- **时间复杂度：** O(n)
- **空间复杂度：** O(1)
- **优点：** 多 Condition 可**定向唤醒**指定线程，扩展到 3 个以上线程（如打印 ABC）时优势明显；语义清晰
- **缺点：** 代码量略多，必须手动 `unlock`

### 解法三：Semaphore（信号量）

> 已有实现见 `OddEvenPrinterSemaphore.java`。

**思路：**
1. 设两个信号量：`oddSemaphore` 初始许可 1，`evenSemaphore` 初始许可 0。
2. 奇数线程先 `oddSemaphore.acquire()` 拿到许可 → 打印 → `evenSemaphore.release()` 把许可交给偶数线程。
3. 偶数线程 `evenSemaphore.acquire()` → 打印 → `oddSemaphore.release()`。
4. 两个信号量像“接力棒”一样在两个线程间来回传递，天然形成交替。

**关键代码：**

```java
public void printOdd() {
    while (number <= maxNumber) {
        oddSemaphore.acquire();
        if (number <= maxNumber && number % 2 != 0) {
            System.out.println("Odd Thread: " + number);
            number++;
        }
        evenSemaphore.release();
    }
}
```

- **时间复杂度：** O(n)
- **空间复杂度：** O(1)
- **优点：** “接力棒”模型最直观，不需要 `while` 重判条件即可保证交替顺序
- **缺点：** 收尾时需注意最后一次 `release` 可能让对方多空转一次（代码里用 `if` 守卫上限规避了误打印）

---

## 四、三种解法对比

| 解法 | 同步机制 | 唤醒方式 | 时间 | 空间 | 优点 | 缺点 |
|------|----------|----------|------|------|------|------|
| 一、wait/notify | synchronized | notifyAll 全部唤醒 | O(n) | O(1) | 最基础、面试常考 | 唤醒不精确 |
| 二、Lock+Condition | ReentrantLock | signal 精确唤醒 | O(n) | O(1) | 可定向唤醒、易扩展多线程 | 代码略多、手动解锁 |
| 三、Semaphore | 信号量许可 | release 传递许可 | O(n) | O(1) | 接力棒模型最直观 | 收尾边界要小心 |

---

## 五、易错点总结

1. **用 `while` 而不是 `if` 判断条件**：`wait/await` 返回后必须重新检查是否轮到自己，防止虚假唤醒。
2. **共享变量 `number` 必须在同步块内访问**：否则存在可见性与竞态。
3. **收尾要唤醒对方**：一方退出循环后若不 `notify/signal`，另一方可能永久阻塞在 `wait/await`，程序挂死。
4. **`unlock` 放在 `finally`**：解法二中无论是否抛异常都要释放锁，否则死锁。
5. **越界保护**：循环条件与打印前都要校验 `number <= maxNumber`，避免打印出超过上限的值。
6. **`InterruptedException` 处理**：捕获后应 `Thread.currentThread().interrupt()` 恢复中断标志，而不是吞掉。
