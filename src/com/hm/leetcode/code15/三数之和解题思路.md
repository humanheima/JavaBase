# LeetCode 15. 三数之和（3Sum）解题文档

## 一、题目描述

给你一个整数数组 `nums`，判断是否存在三元组 `[nums[i], nums[j], nums[k]]` 满足 `i != j`、`i != k` 且 `j != k`，同时还满足 `nums[i] + nums[j] + nums[k] == 0`。请返回所有和为 `0` 且**不重复**的三元组。

注意：答案中不可以包含重复的三元组。

**示例：**

| 输入 | 输出 | 说明 |
|------|------|------|
| `nums = [-1,0,1,2,-1,-4]` | `[[-1,-1,2],[-1,0,1]]` | 两组和为 0 的三元组 |
| `nums = [0,1,1]` | `[]` | 没有和为 0 的三元组 |
| `nums = [0,0,0]` | `[[0,0,0]]` | 唯一的三元组 |

**提示：**
- `3 <= nums.length <= 3000`
- `-10^5 <= nums[i] <= 10^5`

**链接：** https://leetcode.cn/problems/3sum/

---

## 二、核心观察 / 关键点

1. **下标不同、但值可以重复**：`[-1,-1,2]` 是合法的，因为用的是两个不同位置上的 `-1`；但**结果集**里不能出现两个内容相同的三元组。
2. **去重是本题最大的坑**：朴素做法很容易得到一堆重复三元组。先**排序**，让相等的数相邻，就能用「跳过相邻相等元素」的方式优雅去重。
3. **固定一个数 + 双指针**：排序后，固定最小的数 `nums[i]`，原问题就退化成「在 `i` 右侧的有序区间里找两数之和为 `-nums[i]`」，这正是**有序数组两数之和**，可用左右双指针 O(n) 解决。
4. **剪枝**：排序后若 `nums[i] > 0`，则其右侧的数都更大，三数之和不可能为 0，可直接 `break` 结束整个循环。

---

## 三、解法

### 解法一：排序 + 双指针（推荐）

**思路：**
1. 先对数组排序。
2. 从左到右固定第一个数 `nums[i]`（`i` 从 0 到 `n-3`）：
   - 若 `nums[i] > 0`，后面只会更大，直接 `break`。
   - 若 `nums[i] == nums[i-1]`（且 `i > 0`），说明这个固定值已枚举过，`continue` 跳过，避免重复。
3. 在 `[i+1, n-1]` 区间用双指针 `left`、`right` 夹逼，令 `sum = nums[i] + nums[left] + nums[right]`：
   - `sum == 0`：记录三元组，然后**同时跳过 `left`、`right` 两侧重复值**，再 `left++ / right--`。
   - `sum < 0`：和偏小，`left++` 增大。
   - `sum > 0`：和偏大，`right--` 减小。

**关键代码片段：**（与源文件 `ThreeSum.java` 一致）

```java
Arrays.sort(nums);
for (int i = 0; i < n - 2; i++) {
    if (nums[i] > 0) break;                       // 剪枝：最小数 > 0 直接结束
    if (i > 0 && nums[i] == nums[i - 1]) continue; // 跳过重复的固定数

    int left = i + 1, right = n - 1;
    while (left < right) {
        int sum = nums[i] + nums[left] + nums[right];
        if (sum == 0) {
            result.add(Arrays.asList(nums[i], nums[left], nums[right]));
            while (left < right && nums[left] == nums[left + 1]) left++;   // 跳过重复 left
            while (left < right && nums[right] == nums[right - 1]) right--; // 跳过重复 right
            left++;
            right--;
        } else if (sum < 0) {
            left++;
        } else {
            right--;
        }
    }
}
```

- **时间复杂度：** O(n²)。外层固定 O(n)，内层双指针 O(n)；排序 O(n log n) 被吞没。
- **空间复杂度：** O(1)（不含排序的栈开销与结果集）。
- **优点：** 高效、去重干净、是本题标准解法。
- **缺点：** 需要先排序，且去重的双指针跳过逻辑要写对。

### 解法二：暴力三重循环 + 去重（仅用于对照验证）

**思路：**
1. 先排序，保证每个三元组内部有序。
2. 三层循环枚举所有 `i < j < k` 组合，和为 0 时构造三元组。
3. 用 `HashSet<List<Integer>>` 去掉内容重复的三元组（排序后内容相同的三元组其 `List` 也相等）。

**关键代码片段：**

```java
Arrays.sort(nums);
Set<List<Integer>> seen = new HashSet<>();
for (int i = 0; i < n - 2; i++)
    for (int j = i + 1; j < n - 1; j++)
        for (int k = j + 1; k < n; k++)
            if (nums[i] + nums[j] + nums[k] == 0) {
                List<Integer> triple = Arrays.asList(nums[i], nums[j], nums[k]);
                if (seen.add(triple)) result.add(triple);
            }
```

- **时间复杂度：** O(n³)。
- **空间复杂度：** O(n)（去重集合）。
- **优点：** 思路最直白，适合作为正确性基准（cross-check 双指针）。
- **缺点：** n 较大时会超时（本题 n 可达 3000，O(n³) ≈ 2.7×10¹⁰，不可接受）。

## 五、易错点总结

1. **去重的位置**：找到解后要**同时**跳过 `left` 右侧、`right` 左侧的重复值，再各走一步；外层固定数也要 `nums[i] == nums[i-1]` 跳过。漏任一处都会产出重复三元组。
2. **跳过时的边界**：`while (left < right && nums[left] == nums[left + 1]) left++;` 必须带 `left < right` 守卫，否则可能越界或指针交叉。
3. **剪枝条件**：`nums[i] > 0` 才能 `break`（不是 `>= 0`，因为 `0` 仍可能与后面的 0 组成 `[0,0,0]`）。
4. **固定数去重必须「向前看」 `nums[i] == nums[i-1]`，不能「向后看」 `nums[i] == nums[i+1]`**：否则会漏掉像 `[-1,-1,2]` 这种合法解。详见下面第六节专题分析。
5. **小数组**：`n < 3` 直接返回空，避免 `n - 2` 越界。
6. **不要把内层两数之和也再排序**：排序整体只做一次，内层依赖整体有序性。

## 六、专题：固定数去重为什么是 `nums[i] == nums[i-1]`

这是本题最反直觉的一处。结论先行：

> 去重要和**前一个** `nums[i-1]` 比，**不能**和**后一个** `nums[i+1]` 比。

### 6.1 两种写法

```java
// ✅ 正确：向前看。当前固定值和前一个相同 -> 这个固定值已枚举过，跳过
if (i > 0 && nums[i] == nums[i - 1]) continue;

// ❌ 错误：向后看。看到后面还有相同的值就跳过当前
if (i < n - 1 && nums[i] == nums[i + 1]) continue;
```

两种写法都「在遇到连续相同的固定值时只保留一个」，区别在于**保留的是第一个还是最后一个**——而这正是成败关键。

### 6.2 反例：`nums = [-1, -1, 2]`（已排序）

唯一的合法解是 `[-1, -1, 2]`，它**同时用到了两个 `-1`**：第一个 `-1` 当固定数，第二个 `-1` 当 `left`。

**✅ 正确写法（向前看）逐步执行：**

| `i` | `nums[i]` | 去重判断 `nums[i]==nums[i-1]` | 动作 | 内层双指针 |
|-----|-----------|------------------------------|------|------------|
| 0 | `-1` | `i>0` 为假，不跳过 | 进入内层 | `left=1(-1)`, `right=2(2)`，`sum=0` → **记录 `[-1,-1,2]`** |
| 1 | `-1` | `nums[1]==nums[0]` 成立 | `continue` 跳过 | — |

结果：`[[-1, -1, 2]]` ✓

关键在于：**第一个 `-1`（i=0）没被跳过**，它当固定数时，内层指针顺手把第二个 `-1` 当成了 `left`，解被成功捕获。等到 i=1 这个重复的固定值时再跳过，只是避免重复记录，不会丢解。

**❌ 错误写法（向后看）逐步执行：**

| `i` | `nums[i]` | 去重判断 `nums[i]==nums[i+1]` | 动作 | 内层双指针 |
|-----|-----------|------------------------------|------|------------|
| 0 | `-1` | `nums[0]==nums[1]` 成立 | `continue` **跳过** | —（解就在这里被错杀） |
| 1 | `-1` | `nums[1]==nums[2]`（`-1==2`）为假 | 进入内层 | `left=2`, `right=2`，`left<right` 不成立，无任何配对 |

结果：`[]` ✗ —— 合法解 `[-1,-1,2]` 被漏掉了！

错在：向后看会**跳过重复段里的第一个**。可第一个固定值恰恰是唯一能在内层把「后续重复值」配进同一个三元组的机会；它被跳过后，等轮到最后一个 `-1`（i=1）时，它右侧已经没有第二个 `-1` 可用了，于是解永远凑不齐。

### 6.3 一句话记忆

- **向前看**：跳过的是重复段的「后来者」，第一个固定值已经把含重复值的解全找过了 → **安全去重**。
- **向后看**：跳过的是重复段的「先行者」，把唯一能配出 `[x, x, y]` 形解的机会扼杀了 → **会漏解**。

> 同理，内层 `left`/`right` 的去重之所以放在「**找到解之后**」再跳过相邻相同值，也是为了先用第一组配对拿到解，再略过后面的重复，逻辑完全一致。
