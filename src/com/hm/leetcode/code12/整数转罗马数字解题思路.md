# LeetCode 12. 整数转罗马数字（Integer to Roman）解题文档

## 一、题目描述

罗马数字包含七种字符：`I`、`V`、`X`、`L`、`C`、`D`、`M`，对应数值如下：

| 字符 | I | V | X | L | C | D | M |
|------|---|---|---|---|---|---|---|
| 数值 | 1 | 5 | 10 | 50 | 100 | 500 | 1000 |

通常小数字在大数字右边（如 `XII = 12`）。但存在 **6 种减法特例**：小数字放在大数字左边表示相减。

给你一个整数（`1 <= num <= 3999`），将其转为罗马数字。

**示例：**

| 输入 | 输出 | 说明 |
|------|------|------|
| `3` | `III` | 1 + 1 + 1 |
| `4` | `IV` | 5 - 1 |
| `9` | `IX` | 10 - 1 |
| `58` | `LVIII` | L(50) + V(5) + III(3) |
| `1994` | `MCMXCIV` | M(1000) + CM(900) + XC(90) + IV(4) |

**链接：** https://leetcode.cn/problems/integer-to-roman/

---

## 二、核心观察

1. **减法特例只有 6 种**：`IV(4)`、`IX(9)`、`XL(40)`、`XC(90)`、`CD(400)`、`CM(900)`。
2. **化繁为简的关键**：把这 6 种减法形式当作「独立的面值符号」，和 `I/V/X/L/C/D/M` 一起组成 **13 个面值**。这样所有数字都能用「从大到小、能减就减」的贪心方式拼出，无需单独判断特例。
3. **数据范围小**：`num <= 3999`，最多 4 位，千位最多 `MMM`，可以直接按数位打表。

13 个面值（从大到小）：

```
1000  900  500  400  100   90   50   40   10    9    5    4    1
 M    CM    D   CD    C    XC    L   XL    X    IX    V   IV    I
```

---

## 三、解法

### 解法一：贪心 + 减法

**思路：**
1. 把 13 个面值（含 4、9、40、90… 等减法形式）按从大到小排成两个数组 `VALUES` / `SYMBOLS`。
2. 从最大面值开始遍历，**当前面值能减就减**，每减一次就把对应符号追加到结果。
3. `num` 减到 0 结束。

```java
private static final int[] VALUES =
        {1000, 900, 500, 400, 100, 90, 50, 40, 10, 9, 5, 4, 1};
private static final String[] SYMBOLS =
        {"M", "CM", "D", "CD", "C", "XC", "L", "XL", "X", "IX", "V", "IV", "I"};

public static String intToRoman(int num) {
    StringBuilder roman = new StringBuilder();
    for (int i = 0; i < VALUES.length && num > 0; i++) {
        while (num >= VALUES[i]) {     // 当前面值能匹配几次就追加几次
            roman.append(SYMBOLS[i]);
            num -= VALUES[i];
        }
    }
    return roman.toString();
}
```

- **时间复杂度：** O(1)（面值固定 13 个，循环次数有上界）
- **空间复杂度：** O(1)
- **优点：** 通用、易懂，把减法特例统一进面值表，无需特判
- **缺点：** 有内层 while 循环

---

### 解法二：硬编码数位查表

**思路：**
1. 观察到每一位（千、百、十、个）的 `0~9` 罗马表示是**固定**的，可以预先打表。
2. 对 `num` 按位拆分，直接从四张表里取出对应字符串拼接，**无任何循环**。

```java
private static final String[] THOUSANDS = {"", "M", "MM", "MMM"};
private static final String[] HUNDREDS  = {"", "C", "CC", "CCC", "CD", "D", "DC", "DCC", "DCCC", "CM"};
private static final String[] TENS      = {"", "X", "XX", "XXX", "XL", "L", "LX", "LXX", "LXXX", "XC"};
private static final String[] ONES      = {"", "I", "II", "III", "IV", "V", "VI", "VII", "VIII", "IX"};

public static String intToRoman2(int num) {
    return THOUSANDS[num / 1000]
            + HUNDREDS[num % 1000 / 100]
            + TENS[num % 100 / 10]
            + ONES[num % 10];
}
```

- **时间复杂度：** O(1)
- **空间复杂度：** O(1)
- **优点：** 没有循环，按位取表，运行最快
- **缺点：** 表是手工硬编码，可读性差、依赖 `num <= 3999` 的范围假设

## 四、解法对比

| 解法 | 核心思路 | 时间 | 空间 | 优点 | 缺点 |
|------|----------|------|------|------|------|
| 一、贪心减法 | 13 面值从大到小能减就减 | O(1) | O(1) | 通用易懂、无需特判 | 有内层循环 |
| 二、数位查表 | 每位 0~9 预打表直接拼 | O(1) | O(1) | 无循环、最快 | 硬编码、依赖范围 |

> 两种解法时间空间都是常数级（因 num 有上界）。面试推荐**解法一**：思路自然、可解释性强；追求极致性能或代码极简时用**解法二**。

---

## 五、易错点总结

1. **遗漏减法特例**：必须把 `IV/IX/XL/XC/CD/CM` 也放进面值表，否则 4、9、40… 会被错误地写成 `IIII` 这类。
2. **面值顺序**：解法一的 `VALUES` 必须严格从大到小排列，贪心才正确。
3. **数位拆分**：解法二取百位是 `num % 1000 / 100`，先取模再整除，顺序不能错。
4. **范围假设**：解法二的 `THOUSANDS` 表只到 `MMM`，仅适用于 `num <= 3999`。
