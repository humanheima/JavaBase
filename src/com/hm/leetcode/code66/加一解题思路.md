# LeetCode 66. 加一（Plus One）解题文档

## 一、题目描述

给定一个由 **整数** 组成的 **非空** 数组所表示的非负整数，在该数的基础上 **加一**。

最高位数字存放在数组的首位，数组中每个元素只存储单个数字。

你可以假设除了整数 0 之外，这个整数不会以零开头。

**示例：**

| 输入 | 输出 | 说明 |
|------|------|------|
| `[1,2,3]` | `[1,2,4]` | 123 + 1 = 124 |
| `[4,3,2,1]` | `[4,3,2,2]` | 4321 + 1 = 4322 |
| `[9]` | `[1,0]` | 9 + 1 = 10 |
| `[9,9,9]` | `[1,0,0,0]` | 999 + 1 = 1000 |

**提示：**
- `1 <= digits.length <= 100`
- `0 <= digits[i] <= 9`

**链接：** https://leetcode.cn/problems/plus-one/

---

## 二、核心观察

1. **不能直接转成整数计算**：题目允许数组长度达 100 位，远超 `long` 的范围（约 18 位）。任何使用 `int` / `long` / `BigInteger` 的方案都会引入不必要的开销或复杂度。
2. **加一的本质是「末位 +1 + 进位传播」**：只有当末位是 9 时才会产生进位，进而可能影响前一位。只有当整个数组都是 9（如 `[9,9,9]`）时，才需要扩容产生新的最高位 1。
3. **从后往前处理是最高效的顺序**：从低位（数组末尾）开始模拟小学加法，遇到非 9 的数字直接 +1 即可返回；只有连续的 9 才会继续向前传播进位。
4. **原地修改优先**：除了全 9 的特殊情况，大多数情况下可以直接在原数组上修改，无需新建数组。
5. **提前终止优化**：一旦某一位加完后 `carry` 变为 0，说明后续更高位不受影响，可以立即返回。

---

## 三、解法

### 解法：从后往前模拟加法 + 进位处理（推荐）

**思路：**

1. 初始化 `carry = 1`，表示我们要执行「加一」操作。
2. 从数组最后一个元素（最低位）开始向前遍历：
   - 计算 `sum = digits[i] + carry`
   - 把 `digits[i]` 更新为 `sum % 10`（本位结果）
   - `carry = sum / 10`（新的进位）
   - 如果 `carry == 0`，说明已经没有进位需要传播，提前 `break`。
3. 遍历结束后：
   - 如果 `carry` 仍为 1，说明原数字全为 9（如 999），需要新建一个长度为 `n+1` 的数组，首位放 1，其余位为 0。
   - 否则直接返回修改后的原数组。

**关键代码：**

```java
public int[] plusOne(int[] digits) {
    int n = digits.length;
    int carry = 1; // 初始进位为1，相当于加一

    for (int i = n - 1; i >= 0; i--) {
        int sum = digits[i] + carry;
        digits[i] = sum % 10;
        carry = sum / 10;
        if (carry == 0) {
            break; // 无进位则提前终止循环
        }
    }

    if (carry == 1) {
        // 需要扩展数组，例如999 -> 1000
        int[] newDigits = new int[n + 1];
        newDigits[0] = 1;
        return newDigits;
    } else {
        return digits;
    }
}
```

- **时间复杂度：** `O(n)`，最坏情况需要遍历整个数组（全 9 时）。
- **空间复杂度：** `O(1)`（除全 9 情况需要 `O(n)` 新数组外，绝大多数情况是原地修改）。
- **优点：** 
  - 只处理必要的位，平均情况下非常快（有提前 break）。
  - 空间利用极高，不依赖大整数类。
  - 代码简洁直观，符合「大整数加法」的标准模拟思路。
- **缺点：** 需要特别处理全 9 扩容的边界情况（但逻辑清晰）。

### 对比：其他不推荐的思路

**思路二：转成字符串或 BigInteger**
- 把数组拼接成字符串 → 加一 → 再转回数组。
- 或者直接使用 `BigInteger`。
- **缺点**：时间和空间复杂度都更高（需要 O(n) 额外字符串空间），且不符合「原地」或高效模拟的考点意图。

**思路三：递归或从前往后**
- 理论可行，但实现更复杂，且难以提前终止，实际效率不如从后向前。

## 五、易错点总结

1. **进位初始值必须是 1**：很多人容易写成 `carry = 0`，然后单独处理最后一位 +1，容易出错。
2. **数组是「高位在前」**：`digits[0]` 是最高位，`digits[n-1]` 是最低位，因此必须从后往前遍历。
3. **全 9 的扩容**：只有当循环结束后 `carry` 仍然是 1 时才新建数组，且新数组首位一定是 1，后面全 0。
4. **原地修改 vs 返回新数组**：当不需要扩容时直接返回原数组引用即可；扩容时必须返回新数组。
5. **提前 break 的正确性**：只有当 `carry` 变为 0 时才安全 break，否则必须继续向前处理进位。
6. **输入 [0] 的情况**：虽然题目说「除了 0 本身不会以 0 开头」，但 [0] + 1 应该返回 [1]，代码能正确处理。
7. **不要试图用 `digits[i] += 1` 后判断是否 >=10**：这种写法在连续多个 9 时容易漏掉进位传播。
