# LeetCode 27. 移除元素（Remove Element）解题文档

## 一、题目描述

给你一个数组 `nums` 和一个值 `val`，你需要 **原地** 移除所有数值等于 `val` 的元素，并返回移除后数组的新长度 `k`。**元素的顺序可以改变**，你不需要考虑数组中超出新长度后面的元素。

**判题标准：** 假设返回了 `k`，那么 `nums` 的前 `k` 个元素应包含所有不等于 `val` 的元素（顺序不要求一致），`nums` 在 `k` 之后的元素无所谓。

**示例：**

| 输入 | 输出 k | 前 k 项 | 说明 |
|------|--------|---------|------|
| `nums=[3,2,2,3], val=3` | `2` | `[2,2]` | 去掉两个 3 |
| `nums=[0,1,2,2,3,0,4,2], val=2` | `5` | `[0,1,3,0,4]`（顺序可不同） | 去掉三个 2 |
| `nums=[1,1,1], val=1` | `0` | `[]` | 全部移除 |
| `nums=[1,2,3], val=4` | `3` | `[1,2,3]` | 无元素被移除 |

**链接：** https://leetcode.cn/problems/remove-element/

---

## 二、核心观察

1. **原地 + O(1) 空间**：不能新开数组，要在原数组上覆盖写入 → **双指针**。
2. **顺序可以改变**：这一点比 LeetCode 26 更宽松，因此除了"保序的快慢指针"，还能用"对撞指针"减少写入次数。
3. **返回的是长度而非数组**：`k` 之后的内容无所谓，只要前 `k` 个是所有非 `val` 元素即可。
4. **无需有序**：本题数组无需排序，元素任意排列都成立（与 26 题的本质区别）。

---

## 三、解法

### 解法一：快慢指针（保持相对顺序，推荐）

**思路：**

1. `slow` 指向下一个待写入位置，初始 `0`。
2. `fast` 从头扫描每个元素。
3. 当 `nums[fast] != val` 时，把它写到 `nums[slow]`，然后 `slow++`；等于 `val` 则跳过。
4. 结束后 `slow` 即新长度，前 `slow` 个元素保持了原始相对顺序。

**关键代码：**

```java
public int removeElement(int[] nums, int val) {
    int slow = 0;
    for (int fast = 0; fast < nums.length; fast++) {
        if (nums[fast] != val) {
            nums[slow] = nums[fast];
            slow++;
        }
    }
    return slow;
}
```

- **时间复杂度：** `O(n)` —— 扫描一遍。
- **空间复杂度：** `O(1)`。
- **优点：** 保持相对顺序、逻辑直观，是最通用的写法。
- **缺点：** 即使元素不需要移动也会发生 `nums[slow]=nums[fast]` 的自我赋值（无害但有冗余写）。

---

### 解法二：对撞指针（不保序，写入更少）

**思路：**

1. `left` 从左、`right` 从右（初始为 `length`）向中间逼近。
2. 若 `nums[left] == val`，用 **末尾元素** `nums[right-1]` 覆盖它，并 `right--`（被换过来的新元素留在原地，下轮继续判断 `left`，所以 `left` 不前移）。
3. 若 `nums[left] != val`，`left++`。
4. 当 `left == right` 时结束，`left` 即新长度。

**为什么更省写入？** 当某个 `val` 出现时，只做 **一次** 覆盖；不像快慢指针那样把后面所有元素逐个前移。适合 **待删元素很少** 的场景。

**关键代码：**

```java
public int removeElementTwoEnds(int[] nums, int val) {
    int left = 0, right = nums.length;
    while (left < right) {
        if (nums[left] == val) {
            nums[left] = nums[right - 1]; // 用末尾覆盖当前
            right--;                      // 末尾收缩
        } else {
            left++;
        }
    }
    return left;
}
```

- **时间复杂度：** `O(n)`。
- **空间复杂度：** `O(1)`。
- **优点：** 写入次数少（每个待删元素只触发一次覆盖），对"删的很少"的数据更优。
- **缺点：** **不保持相对顺序**（题目允许），且覆盖后 `left` 不前移这点容易写错。

---

## 四、解法对比表

| 解法 | 核心思路 | 时间 | 空间 | 是否保序 | 适用场景 |
|------|----------|------|------|----------|----------|
| 一、快慢指针 | 非 val 元素依次前移写入 | O(n) | O(1) | 是 | 通用首选 |
| 二、对撞指针 | val 处用末尾覆盖 | O(n) | O(1) | 否 | 待删元素很少 |

**推荐：** 默认用快慢指针（直观、保序）；若强调减少写入或待删极少，可用对撞指针。

---

## 五、易错点总结

1. **对撞指针覆盖后 `left` 不能前移**：因为从末尾换过来的元素还没判断，可能仍等于 `val`，必须在原地再判一次。
2. **`right` 的初值与边界**：用 `right = nums.length`，循环条件 `left < right`，覆盖时取 `nums[right-1]`；不要把 `right` 初始化成 `length-1` 后又用错下标。
3. **返回长度不是下标**：快慢指针返回 `slow`（它本身就是已写入个数）；对撞指针返回 `left`。
4. **空数组**：`length == 0` 时两种解法都自然返回 0，无需特判。
5. **顺序要求**：本题允许乱序，但如果题目改成"必须保序"，则只能用快慢指针。
6. **与 LeetCode 26 区分**：26 题要求有序去重、比较相邻元素；27 题是删指定值、无序也成立，别混用条件。
